# F.21 Architecture: Scouting Data Pipeline
# Status: Architecture only — students should build the scouting app themselves

## What It Does

Defines a standard schema for scouting data so that The Engine's pre-match
strategist (F.13), post-match analyzer (F.14), and performance tracker (F.24)
can all consume the same data format.

The scouting APP is a student project — they learn more building it than using
a pre-built one. This doc defines the contract between the app and The Engine.

---

## Data Schema

### Per-Match Scout Record

One record per robot per match, entered by a human scout in the stands:

```json
{
  "meta": {
    "event_key": "2028cmptx",
    "match_key": "qm42",
    "match_type": "qual",
    "team": 2950,
    "scouted_by": "alex",
    "timestamp": "2028-03-15T14:23:00Z"
  },
  "auto": {
    "starting_position": "left",
    "pieces_scored": 3,
    "pieces_attempted": 3,
    "mobility": true,
    "notes": ""
  },
  "teleop": {
    "pieces_scored": 8,
    "pieces_attempted": 10,
    "preferred_scoring_location": "left_high",
    "played_defense": false,
    "defense_effectiveness": null,
    "was_defended": true,
    "scoring_while_defended": 3
  },
  "endgame": {
    "climb_attempted": true,
    "climb_succeeded": true,
    "climb_time_s": 2.1,
    "climb_level": "deep",
    "parked": false
  },
  "qualitative": {
    "driver_skill": 4,
    "robot_speed": 5,
    "intake_reliability": 4,
    "overall_threat": 5,
    "notes": "Very fast cycles, strong left-side scorer"
  }
}
```

**Qualitative scores:** 1-5 scale (1=poor, 3=average, 5=elite).

## File Format & Storage

```
scouting-data/
├── events/
│   └── 2028cmptx/
│       ├── raw/
│       │   ├── qm01_2950_alex.json    (one file per scout record)
│       │   ├── qm01_254_jordan.json
│       │   └── ...
│       ├── summaries/
│       │   ├── 2950.json              (aggregated team summary)
│       │   ├── 254.json
│       │   └── ...
│       └── event_meta.json            (team list, schedule, rankings)
└── tools/
    └── aggregate.py                   (raw → summaries)
```

- **Raw files:** Written by the scouting app (phone, tablet, laptop — whatever the students build)
- **Summaries:** Generated by `aggregate.py` after each round of matches
- **Consumers:** F.13 (pre-match strategist) reads summaries; F.14 (post-match) reads raw; F.24 (cross-event tracker) reads all events

## Integration Points

### F.13 Pre-Match Strategist

Reads team summaries for all 6 robots in the upcoming match. Feeds them into
the prompt template as the `alliance_partners` and `opponents` fields.

```python
# In the pre-match pipeline:
our_team = load_summary("scouting-data/events/{event}/summaries/2950.json")
partners = [load_summary(f".../{t}.json") for t in alliance_partner_teams]
opponents = [load_summary(f".../{t}.json") for t in opponent_teams]
prompt_input = build_strategist_input(our_team, partners, opponents, match_info)
```

### F.14 Post-Match Analyzer

Reads OUR raw scout data + AdvantageKit log. Compares human observations to
robot telemetry:
- Scout says 8 pieces scored, CycleTracker logged 7 → investigate the gap
- Scout says climb took 2.1s, telemetry says 2.4s → scout timer may be off

### F.24 Performance Tracker

Reads summaries across multiple events. Tracks trends:
- Is our auto success rate improving? (Event 1: 80%, Event 2: 88%, Event 3: 95%)
- Are cycle times getting faster or slower?
- Which alliance partners did we perform best with?

## What Students Build vs What The Engine Provides

| Component | Who Builds | Notes |
|-----------|-----------|-------|
| Scouting app (UI) | Students | Phone/tablet app, Google Form, paper + spreadsheet — their choice |
| Data schema | The Engine (this doc) | Students must output this JSON format |
| aggregate.py | The Engine (F.21) | Simple script, could also be a student project |
| Pre-match strategist | The Engine (F.13) | Consumes summaries |
| Post-match analyzer | The Engine (F.14) | Consumes raw + logs |
| Performance tracker | The Engine (F.24) | Consumes cross-event summaries |

## Scouting App Recommendations

The students can build whatever they want as long as it outputs the JSON schema.
Common approaches in FRC:

| Approach | Pros | Cons |
|----------|------|------|
| **Google Forms + Sheets** | Zero code, works offline on phones | Annoying to export to JSON, slow |
| **Custom web app** | Full control, runs on any device | Needs a student who knows web dev |
| **Paper + data entry** | Works with zero technology | Slowest, error-prone |
| **The Blue Alliance API** | Free OPR/DPR/CCWM data | No qualitative data, delayed |

**Recommendation:** Start with Google Forms for qualitative scouting. Pull
quantitative data (OPR, match scores) from The Blue Alliance API. Merge both
into the schema above using a Python script.

## Minimum Viable Implementation

If the students don't build a scouting app in time, the absolute minimum is:

1. One student per match writes notes on paper using the schema fields as a checklist
2. Between matches, type it into a shared Google Sheet
3. Before playoffs, run a script to convert the Sheet to JSON summaries
4. Feed summaries to F.13 pre-match strategist

This is ugly but functional. Better scouting infrastructure can come later.
